# 前后端功能对照表

## 一、认证与登录（Auth）

### 1.1 两阶段登录流程

| 步骤 | 前端 API | 后端接口 | 说明 |
|------|---------|---------|------|
| Step 1 | `POST /auth/login` | `AuthController.login` | 仅校验账号/密码/验证码，返回 `PreAuthLoginResponse`（含 `preAuthToken` + `shops[]`，不签发正式 JWT） |
| Step 2 | `POST /auth/confirm-shop` | `AuthController.confirmShop` | 凭 `preAuthToken + shopId` 确认店铺，签发正式 JWT，写入在线会话 |
| 路由守卫 | 前端路由 `beforeEach` | — | 只认正式 JWT + `currentShopId`，有预登录票据才允许进 `?step=shop` |

### 1.2 登录态与预登录态

| 状态 | 存储位置 | 生命周期 | 内容 |
|------|---------|---------|------|
| 预登录态 | `sessionStorage` | 账号验证成功后、确认店铺前（10min TTL） | `preAuthToken` + `pendingShops[]` |
| 正式登录态 | `localStorage` | 确认店铺成功后长期有效（JWT 有效期） | `nexus_token`（JWT） + `nexus_current_shop_id` |

### 1.3 登录后操作

| 操作 | 前端方法 | 后端接口 | 说明 |
|------|---------|---------|------|
| 登出 | `userStore.logout()` | `POST /auth/logout` | 注销 Redis 在线会话，清空本地存储 |
| Token 刷新 | `request.ts` interceptor 自动触发 | `POST /auth/refresh` | JWT 过期前 5 分钟自动刷新（Rotation） |
| 切换店铺 | `userStore.switchShop(id)` | `PUT /auth/switch-shop` | 已登录用户换店铺，签发新 JWT |
| 刷新用户信息 | `userStore.fetchUserInfo()` | `GET /system/user/info` | 返回 `{ profile, menus, latestNoticeTitle }` |
| 获取店铺列表 | `userStore.getPendingShops()` / `loadPendingShopsIntoTree()` | — | 预登录阶段从 sessionStorage 读；登录后从 `profile.accessibleShopIds` 读 |

### 1.4 后端关键 DTO

| DTO | 路径 | 用途 |
|-----|------|------|
| `LoginRequest` | `AuthDtos.LoginRequest` | Step1 请求：username, password, captcha, captchaKey |
| `PreAuthLoginResponse` | `AuthDtos.PreAuthLoginResponse` | Step1 响应：preAuthToken, tenantId, recommendedShopId, shops, expiresInSeconds, requiresShopSelection |
| `ConfirmShopRequest` | `AuthDtos.ConfirmShopRequest` | Step2 请求：preAuthToken, shopId |
| `LoginResponse` | `AuthDtos.LoginResponse` | Step2/switchShop 响应：accessToken, tokenType, tenantId, currentShopId, currentOrgId, dataScope, accessibleShopIds, accessibleOrgIds |
| `UserInfoResponse` | `UserInfoDtos.UserInfoResponse` | `GET /system/user/info` 响应：profile (UserProfile), menus (MenuNode[]), latestNoticeTitle |
| `UserProfile` | `UserInfoDtos.UserProfile` | 用户信息片：userId, username, realName, avatarUrl, tenantId, currentShopId, currentOrgId, dataScope, accessibleShopIds, accessibleOrgIds |

## 二、菜单与路由

| 功能 | 前端实现 | 后端实现 |
|------|---------|---------|
| 菜单树 | `GET /system/user/info` 返回 `menus: MenuNode[]` | `SystemUserInfoService.queryMenuTree()` 按 shopId 查角色→菜单 |
| 动态路由注册 | `userStore.addAccessibleRoutes()` → `router.addRoute('Layout', ...)` | — |
| 一级模块侧边栏 | `AppSidebar.vue` 从 menus 提取一级节点 | 后端 menus 第一层为一级模块 |
| 二级 Tab 栏 | `SubTabs.vue` 从 menus 提取二级菜单 | — |
| 快捷操作（Dashboard） | `dashboard/index.vue` 递归收集所有 `component` 叶子菜单 | — |
| Dashboard 用户信息 | `userStore.profile?.realName / username` | `profile.realName / profile.username` |
| 当前店铺（Dashboard/Topbar） | `userStore.currentShop?.shopName` | `profile.currentShopId` → `shops` 映射 |

## 七、当前已知不一致/待办

| # | 问题 | 影响 |
|---|------|------|
| 1 | `variables.css` 与 `tokens.css` 均定义 layout 变量，后者覆盖前者 | `--sidebar-width` 实际为 72px，`--header-height` 实际为 64px |
| 2 | `html, body, #app` 使用 `height: 100%` 而非 `100vh` | iOS 等场景下可能出现滚动条问题 |
| 3 | 员工页面（`oa/employee`）的 `.employee-page` 缺少 `height: 100%` | 左右两栏不能自动撑满可用高度 |
| 4 | `el-dialog` 未显式设置 `append-to-body` | 弹窗遮罩可能被父容器裁切 |
| 5 | `el-tree-select` 在 login 页的 `clearable` 与 placeholder 共存问题 | 体验问题 |
